1 · What Ntole is

Ntole is a ride-hailing progressive web app targeted at Malawi. A single Next.js 15 application ships three role-scoped surfaces from one codebase, all sharing a Supabase backend:

  • Rider — the public-facing hailing experience: enter pickup and destination, see real-time fare quotes per ride tier, request a ride, track the assigned driver on a live map, and pay.
  • Driver — onboarding/compliance wizard, online/offline toggle, incoming ride requests, in-trip state machine (arrive → start → end), and earnings.
  • Admin — operational console: users, drivers (including review of submitted applications), rides, dynamic pricing tiers, and a fleet live-map.

The fare model is the standard Uber-style equation, applied per tier:

fare = max(min_fare, (base + per_km × distance + per_min × duration) × surge)

Tiers, rates, and surge multipliers are stored in ride_tiers and are admin-editable.

2 · Technology stack

Next.js 15 (App Router) + React 19, written in TypeScript.

Tailwind CSS with a Material-3-inspired token system (custom tailwind.config.ts). Light + dark themes via a ThemeProvider.

Plus Jakarta Sans (single sans family across display, body, and labels).
Backend
Supabase — Postgres for data, GoTrue for auth, Realtime for live channels, Storage for driver document and avatar uploads.
Authorization

Maps

Payments
Provider-abstracted: PaymentProvider interface with Stripe and a mock implementation selected by PAYMENT_PROVIDER.
PWA
Service worker registration + install prompt for "Add to home screen". Manifest in public/manifest.webmanifest.
Hosting
Render.com (auto-deploys from the main branch).
Testing
Vitest (currently exercising the pricing engine — see lib/pricing/fare.test.ts).

4 · Authentication & access control

Sign-in flow

  • Email + password via Supabase Auth (signInWithPassword / signUp).
  • Confirmation emails (when enabled in Supabase) come back through app/auth/callback/route.ts, which exchanges the code for a session cookie.
  • Three seeded accounts ship in supabase/seed.sql for QA (rider, driver, admin).

Middleware

middleware.ts runs on every request and:

  1. Refreshes the Supabase session cookie (updateSession).
  2. Lets public paths through: /, /login, /rider, and anything under /auth/.
  3. Otherwise requires a session — redirects to /login?next=… if absent.
  4. Reads the user's role from profiles and gates top-level segments via canAccess() in lib/auth/routing.ts.
  5. Forces unapproved drivers through /driver/apply before they can reach the rest of the driver surface.

Server components

Protected server components call requireRole(...roles) (in lib/auth/session.ts) which redirects mismatched users to their own home.

Row Level Security

Every table has RLS enabled. Representative policies:

  • profiles_self_read — a user reads their own row; admins read all.
  • profiles_ride_partner_read — riders and drivers can read each other's basic profile, only for rides they share.
  • rides_participant_read / update — only the rider, the driver, or an admin.
  • messages_participant_* — chat messages are visible only to the two ride participants.
  • favorite_drivers_self_* — riders can favourite drivers, only manage their own list.
  • "avatars self write" on storage.objects — users upload only into avatars/<uid>/….

Atomic ride acceptance uses a SECURITY DEFINER RPC (accept_ride(ride)) so two drivers can't both win the same request.

5 · Database schema

Three migrations under supabase/migrations/. All idempotent.

Core tables (migration 0001)

TableWhat it stores
profilesOne row per auth.users row. role (rider / driver / admin), full_name, phone, avatar_url, rating, trip_count, safety_rating.
ride_tiersPricing tiers (Standard, Comfort, Lux, etc.). Base fare, per-km, per-minute, min fare, currency, surge multiplier, seat count, ordering, active flag.
vehiclesDriver-owned vehicles. Make, model, plate, color, seats, tier.
driversDriver-specific data. Linked vehicle, status (offline / online / on_trip), current GPS coords, last-seen timestamp, license number, verification flag.
ridesThe lifecycle of a ride. Foreign keys to rider / driver / vehicle / tier. Status enum, pickup & drop coords + address, quoted vs actual distance & duration, fare, currency, surge multiplier, and a full set of timestamps (requested, accepted, arrived, started, completed, cancelled).
messagesIn-ride chat between rider and driver.
paymentsOne row per payment intent. Provider, provider intent id, amount, status (pending / authorized / paid / failed / refunded).
ratingsOne-per-ride-per-rater 1–5 star rating + optional comment. A trigger recomputes the recipient's profiles.rating average.
app_configSingle-row global settings (default currency, match radius, ping interval, surge).

Profile extensions (migration 0002)

  • profiles.home_lat / home_lng / home_address — saved "Home" pickup pin used by the rider quick-chip rail.
  • favorite_drivers (rider_id, driver_id) — riders can star drivers they've ridden with.
  • storage.buckets.avatars — public read, write only into <uid>/…. Powers self-served profile picture uploads.

Driver compliance (migration 0003)

Adds onboarding, moderation, and complaint primitives:

  • drivers gains approval_status (draft → submitted → approved / rejected / banned), normalized national ID + license fields, document paths in Storage, requested vs admin-assigned tier, rating overrides, warning count, ban metadata.
  • banned_identifiers — reuse-resistant ban registry keyed by national ID + license.
  • complaints — rider/driver complaints with category and status.
  • driver_moderation_log — audit trail of every approve / reject / warn / ban / unban / tier change / rating override / verify action.

Key Postgres helpers

  • haversine_km(...) — great-circle distance in km.
  • find_nearby_drivers(origin_lat, origin_lng, tier, radius, limit) — returns the N closest online drivers in a tier within a radius.
  • accept_ride(ride) — atomic claim used by PATCH /api/rides/[id].
  • is_admin() — predicate used by every admin-gated RLS policy.
  • Triggers: touch_updated_at on all mutable tables, handle_new_user seeds a profile when a Supabase auth user is inserted, recompute_rating updates running averages, increment_trip_counts on completion.

12 · Realtime, maps & geolocation

Realtime channels

  • Driver location — the driver client publishes its GPS every NEXT_PUBLIC_DRIVER_LOCATION_INTERVAL_MS (5 s default) while online. The rider's active-trip screen subscribes to the driver's channel so the marker moves smoothly.
  • Ride status — the rider (and driver) subscribe to their ride row in rides; status transitions trigger panel swaps without a reload.
  • Admin fleet — AdminLiveMap subscribes to every drivers row and plots online drivers in realtime.
  • In-ride chat — messages table is in the Realtime publication; new rows appear instantly on both sides.

Maps

  • Basemap — CartoDB Dark Matter (dark theme) or Voyager (light theme). Theme-aware tile URL inside RideMap.
  • Markers — custom div-icons drawn in the brand neon-green. Pickup pin, destination pin, animated car icon for the assigned driver, and small fleet dots for the admin view.
  • Route line — three concentric strokes for a glow effect over the encoded polyline returned by OSRM.
  • Recentering — RideMap exposes recenterTo (smooth flyTo without rendering a marker) and markersHidden (background-only use). The rider home uses both to follow the user's GPS coordinates.

Geolocation

  • useGeolocation hook prompts for browser permission once on the rider home.
  • On success, the page reverse-geocodes the fix through /api/geocode/reverse and auto-fills the Pickup field with the resolved address (falling back to raw coordinates if the lookup misses).
  • An error boundary wraps the rider home so any failure surfaces a friendly retry card rather than crashing the page.

13 · Payments

Payments are abstracted so we can swap providers without touching the UI.

  • PaymentProvider interface declares createIntent, verifyWebhook, etc.
  • Stripe is the production implementation (Payment Intents API).
  • Mock auto-completes intents for local development and demos. Selected when PAYMENT_PROVIDER=mock.
  • The rider's completed-ride panel calls POST /api/payments/intent as soon as the ride hits completed.
  • The Stripe webhook (POST /api/payments/webhook) updates payments.status to paid, failed, or refunded.

Production note. Stripe supports MWK but merchant onboarding in Malawi is limited. The PaymentProvider interface is designed so a local provider (e.g. PayChangu, Airtel Money) can be added by implementing one file.

16 · Glossary

Surface
A role-scoped section of the app — rider, driver, or admin. Each surface has its own layout, navigation, and Tailwind tokens.
Tier
A pricing tier in ride_tiers (e.g. Standard / Comfort / Lux). Drives both rider-side fare quotes and driver-matching.
RLS
Row-Level Security — Postgres policies enforced on every query made with the anon key.
RPC
A Postgres function callable via Supabase. We use SECURITY DEFINER functions for atomic operations (e.g. accept_ride) and admin predicates (is_admin).
Surge
Multiplier applied on top of the base fare formula. Configurable per tier and globally via app_config.
Approval status
State of a driver application — draft → submitted → approved / rejected / banned. Middleware blocks unapproved drivers from the dashboard.
Moderation log
Append-only audit trail of every admin action against a driver (approve, reject, warn, ban, unban, change tier, override rating, verify).
